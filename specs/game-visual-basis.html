Purpose

Produce every visual asset the CheenaSim game needs to run: a polished house background, eight full-screen room backgrounds (six house rooms + work + driving), per-room interactive prop sprites, and expanded character spritesheets covering all animation poses referenced in the activity/minigame system plan. Alongside the assets, establish a robust TypeScript asset manifest and Phaser lazy-load pipeline so any future minigame can declare its asset dependencies in one place and have them loaded automatically — with zero manual wiring in BootScene.

Problem

The game currently ships with three assets: house.png, sim-violet.png, and sim-cyan.png. The character spritesheets are 128×48 (four idle frames only). There are no room close-up backgrounds, no prop sprites for POI interactions, and no animation frames for sitting, walking, reading, or driving. The activity/minigame system plan (phases 4–7) requires all of these assets before a single minigame can run. There is also no pipeline — adding a new minigame asset today means manually editing BootScene.preload() and remembering which file goes where.

Solution

Generate all assets using OpenAI gpt-image-1 (via the project's generate_gpt_image.py / edit_gpt_image.py scripts) combined with ImageMagick v7 (magick) for resize, palette reduction, and spritesheet stitching. Assets are organized into a predictable directory structure. A src/config/asset-manifest.ts registry maps every room and activity to its required files. A generic PreloadScene lazy-loads activity-specific assets before launching any minigame, so BootScene only ever loads the minimum needed for the main house view. Character spritesheets are extended in-place from 4 frames (idle) to 22 frames covering all animation poses the minigame system needs.

// Central registry: room → background key + prop asset keys.
// Add new props here — PreloadScene picks them up automatically.

export interface RoomAssets {
  bg:    string;      // Phaser texture key for the room plate background
  props: string[];   // Phaser texture keys for interactive prop sprites
}

export const ROOM_ASSETS: Record<string, RoomAssets> = {
  bedroom:     { bg: 'rooms/bedroom-bg',     props: ['props/bedroom-bed',          'props/bedroom-dresser'] },
  bathroom:    { bg: 'rooms/bathroom-bg',    props: ['props/bathroom-bathtub',     'props/bathroom-sink'] },
  kitchen:     { bg: 'rooms/kitchen-bg',     props: ['props/kitchen-stove',       'props/kitchen-fridge',   'props/kitchen-table'] },
  living_room: { bg: 'rooms/living-room-bg', props: ['props/living-room-sofa',    'props/living-room-tv',   'props/living-room-table', 'props/living-room-magazine-pile'] },
  garden:      { bg: 'rooms/garden-bg',      props: ['props/garden-door'] },
  study:       { bg: 'rooms/study-bg',       props: ['props/study-desk',          'props/study-bookshelf'] },
  work:        { bg: 'rooms/work-bg',        props: ['props/work-desk'] },
  driving:     { bg: 'rooms/driving-bg',    props: [] },
};

// Returns all asset keys needed for a given room (bg + props).
export function getAssetsForRoom(room: string): string[] {
  const r = ROOM_ASSETS[room];
  if (!r) return [];
  return [r.bg, ...r.props];
}

// Maps an asset key to its file path (relative to public/).
export function assetPath(key: string): string {
  return `assets/${key}.png`;
}

1.3. PreloadScene — src/scenes/PreloadScene.ts

A generic lazy-load scene. Minigame scenes start via PreloadScene when they need room-specific assets not loaded at boot. This keeps BootScene lean.

import Phaser from 'phaser';
import { assetPath } from '../config/asset-manifest';

export interface PreloadSceneData {
  assets:      string[];   // asset keys to load (see asset-manifest)
  targetScene: string;    // scene to start after loading
  targetData?: object;   // forwarded to targetScene.init()
}

export class PreloadScene extends Phaser.Scene {
  private data!: PreloadSceneData;

  constructor() { super({ key: 'PreloadScene' }); }

  init(data: PreloadSceneData): void { this.data = data; }

  preload(): void {
    for (const key of this.data.assets) {
      if (!this.textures.exists(key)) {
        this.load.image(key, assetPath(key));
      }
    }
    this._drawProgress();
  }

  create(): void {
    this.scene.start(this.data.targetScene, this.data.targetData);
  }

  private _drawProgress(): void {
    const bar = this.add.graphics();
    this.load.on('progress', (v: number) => {
      bar.clear();
      bar.fillStyle(0x8b7ff7);
      bar.fillRect(180, 310, 600 * v, 20);
    });
  }
}

Notes

OPENAI_API_KEY Required All asset generation commands use uv run .../generate_gpt_image.py, which requires OPENAI_API_KEY to be set in the shell environment. Export it before running any generation step: export OPENAI_API_KEY=sk-.... The key does not need to be committed to the repo.
Sprite Quality Trade-off AI-generated art downscaled to 32×48 will look rough compared to hand-crafted pixel art. This is intentional — the primary goal of this plan is to establish functional animation frames that unblock the minigame system. Visual polish of individual frames can be done in a follow-up pass using Aseprite or similar tools. The pipelines defined here serve as the repeatable template for that refinement too.

Why OpenAI for Game Assets?

The design principle is speed over polish for the initial pass. Every generated asset serves as a functional placeholder that can be refined later without changing any code — the manifest keys, Phaser texture keys, and animation frame maps are all stable. Replacing bedroom-bg.png with hand-drawn pixel art in the future requires zero code changes.

Minigame Asset Pipeline — Adding New Assets

When a new minigame needs its own props or background variants:

  1. Add the new asset to ROOM_ASSETS[roomName].props in asset-manifest.ts
  2. Generate the PNG using the prop template from Phase 4.8 and drop it in public/assets/props/
  3. The minigame scene starts via PreloadScene with getAssetsForRoom('room_name') — it will pick up the new prop automatically
  4. No changes to BootScene required

Driving Background Tileability

The driving background is used as a Phaser.GameObjects.TileSprite in DrivingScene. If the generated image has a visible seam at the horizontal edges, use ImageMagick's offset trick to create a seamless tile:

magick digital-world/public/assets/rooms/driving-bg.png \
  -roll +480+0 \
  /tmp/driving-bg-offset.png
# Inspect /tmp/driving-bg-offset.png to find and fix the seam manually,
# then re-save as driving-bg.png.
